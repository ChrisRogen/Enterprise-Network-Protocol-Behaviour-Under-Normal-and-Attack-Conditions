What DHCP Does

DHCP automates client network configuration. Instead of manually assigning IP addresses and gateway settings to every host, a DHCP server leases configuration to clients. In a normal network, this process reduces configuration errors and allows new hosts to join the network quickly.

Why DHCP Can Be Attacked

DHCP depends on client trust during the early configuration stage. A rogue DHCP server can reply to a client before or alongside the legitimate server. If the client accepts the rogue offer, the attacker may assign a malicious gateway, DNS server, or incorrect network configuration.

Protocol Understanding

Normal DHCP DORA vs Rogue DHCP Behaviour

DHCP is useful because it automates network configuration, but it becomes risky when a rogue DHCP responder is present. The difference between a trusted server and a rogue server is important because clients may accept network settings before they can independently verify whether the responder is legitimate.

Normal DHCP DORA Sequence

In normal DHCP behaviour, the client starts without a valid IP address and broadcasts a DHCP Discover message. The legitimate DHCP server responds with an Offer. The client sends a Request to accept the proposed configuration, and the server completes the process with an ACK. This gives the client a valid IP address and supporting network settings.

Rogue DHCP Attack Sequence

In rogue DHCP behaviour, an attacker operates an unauthorised DHCP service. When the client broadcasts a Discover message, the rogue server can send a competing Offer. If the client accepts the rogue configuration, it may receive a malicious gateway, DNS server, or incorrect IP settings that redirect traffic or disrupt normal communication.

Normal vs Attack Difference

The key difference is trust. In the normal path, the client receives configuration from the correct branch DHCP server. In the attack path, the attacker competes with the legitimate server and attempts to control the client’s network configuration before normal communication begins.

Security Analysis

What the Attack Shows

The rogue DHCP scenario shows that host configuration can be attacked before the user begins normal network activity. If the attacker controls DHCP settings, the victim can be directed to a malicious gateway, given a hostile DNS server, or placed into a misconfigured network state. This makes DHCP abuse especially dangerous in local networks where clients trust the first usable configuration they receive.

Defensive Thinking

How Rogue DHCP Risk Can Be Reduced

Practical mitigations include DHCP snooping, switch port security, trusted uplink configuration, network access control, VLAN segmentation, static addressing for critical infrastructure, and monitoring for unexpected DHCP offers. Wireshark filters such as bootp or dhcp can help identify competing DHCP servers during investigation.

Reproduction Package

Recreate the DHCP Practical on Another Local System

Open DHCP Google Drive Folder Open DHCP Instruction Manual PDF

The DHCP reproduction package should contain the DHCP packet capture files, the OVA file, and the instruction manual required to recreate the practical environment. The instruction manual PDF provides the full step-by-step process from importing the Ubuntu OVA into VirtualBox through to running MiniEdit/Mininet, capturing DHCP traffic in Wireshark, executing normal DHCP DORA testing, running rogue DHCP testing, saving evidence, and comparing the practical output with the website simulation.

1

Download Artefacts

Open the DHCP Google Drive folder and download the OVA file, DHCP PCAP files, and the instruction manual. Keep all files together in one local project folder so the VM, packet captures, screenshots, and recreation steps remain organised.

2

Read the Instruction Manual

Open dhcp-practical-recreation-instruction-manual.pdf before starting the practical. Follow the manual in order because it explains the required software, OVA import process, topology setup, Wireshark filters, normal DHCP DORA commands, and rogue DHCP testing commands.

3

Import the Ubuntu OVA

In Oracle VirtualBox, choose File → Import Appliance, select the DHCP OVA file, and complete the import. After import, start the Ubuntu VM and confirm that the desktop, terminal, MiniEdit/Mininet, and Wireshark are available.

4

Run the Normal DHCP Test

Use MiniEdit/Mininet to start the DHCP topology. Start the legitimate DHCP service, release any old client lease, request a new DHCP lease from the client, and capture the Discover, Offer, Request, and ACK sequence in Wireshark using filters such as bootp or dhcp.

5

Run the Rogue DHCP Test

Follow the attack section in the manual to start a rogue DHCP responder from the attacker host. Capture the competing DHCP offer in Wireshark and compare the server identifier, offered IP address, router option, and DNS option against the legitimate DHCP response.

6

Compare with Website

Return to this DHCP page and run the predefined normal and attack simulations. Compare the website packet animation and event logs with the Wireshark DHCP evidence so the website, practical lab, and packet captures are clearly connected.
