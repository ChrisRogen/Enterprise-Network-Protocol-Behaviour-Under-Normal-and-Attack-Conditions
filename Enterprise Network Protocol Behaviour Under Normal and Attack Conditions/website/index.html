Enterprise Network Protocol Behaviour Under Normal and Attack Conditions

This project demonstrates how key enterprise network protocols behave when they operate correctly and how the same protocols can be abused during cybersecurity attacks. The core focus is ARP, DHCP, and DNS because these protocols are essential for local network communication, automatic host configuration, and name resolution, but they can also be targeted through spoofing, rogue service responses, and traffic redirection.

The website provides an interactive learning layer where packet movement, validation messages, and real-time output logs are shown visually. The practical layer was completed in Ubuntu using MiniEdit/Mininet, and the evidence layer was captured using Wireshark. Together, these components show that the project is not only a visual animation, but also a technically validated cybersecurity demonstration.

Project Purpose

Project Mission

The purpose of this project is to make network protocol behaviour easier to understand by combining visual packet animation with real technical validation. Instead of only describing ARP, DHCP, and DNS in theory, the website shows how packets move across an enterprise-style topology and how the same behaviours can be proven through Ubuntu-based practical testing. This supports deeper cybersecurity learning because the project does not only show that a protocol works; it also shows how attackers can misuse normal protocol trust relationships and how abnormal behaviour can be identified from packet evidence.

Technical Validation

Simulation Backed by Real Packet Evidence

Each completed protocol area is designed around two layers of evidence. The first layer is the website simulation, where users can trigger predefined and manual normal or attack scenarios and observe packet movement, validation messages, and real-time output logs. The second layer is the Ubuntu MiniEdit/Mininet practical environment, where normal and attack behaviour was reproduced and captured using Wireshark.

This connection between simulation and packet capture strengthens the project because the visual explanation is supported by real protocol behaviour rather than being only a decorative animation. The final demonstration can therefore explain the website simulation, the Ubuntu lab setup, and the Wireshark packet evidence as one connected workflow.

Website Simulator

The website provides the interactive learning layer. It allows users to run normal and attack demonstrations, enter manual source and destination values, and receive validation feedback when a path is valid, invalid, or unsupported.

Ubuntu Practical Lab

The Ubuntu MiniEdit/Mininet lab provides the practical testing layer. Client, server, gateway, and attacker roles were used to reproduce the protocol behaviour in a controlled network environment.

Wireshark Evidence

Wireshark provides the packet-level verification layer. Packet captures confirm ARP requests and forged replies, DHCP DORA and rogue offers, and legitimate versus rogue DNS responses.

Completed Core Scope

Implemented Protocol Simulations

ARP, DHCP, and DNS are the completed core protocol areas in this project. Each protocol page includes a visual topology, predefined normal behaviour, predefined attack behaviour, manual input validation, real-time packet output, and reset functionality for repeated demonstration.

ARP Completed

ARP: Address Resolution and Spoofing

The ARP simulation demonstrates how a host resolves an IP address into a MAC address inside a local network. In normal behaviour, a client sends an ARP request and receives a legitimate ARP reply from the correct destination or gateway. In attack behaviour, a malicious host sends forged ARP replies so the victim stores an incorrect IP-to-MAC mapping. This allows traffic to be redirected through the attacker, demonstrating how ARP spoofing can support man-in-the-middle activity.

DHCP Completed

DHCP: Lease Allocation and Rogue DHCP

The DHCP simulation demonstrates the normal DORA process: Discover, Offer, Request, and Ack. In normal behaviour, a client requests network configuration and receives a valid IP address, gateway, DNS server, and lease information from the correct branch DHCP server. In attack behaviour, a rogue DHCP responder competes with the legitimate server and can provide malicious or incorrect configuration, such as a hostile gateway or DNS server.

Open DHCP Page
DNS Completed

DNS: Name Resolution and Rogue Response

The DNS simulation demonstrates how a client resolves a domain name into an IP address by querying a DNS server. In normal behaviour, the legitimate DNS server returns the correct address for the requested domain. In attack behaviour, a rogue DNS responder provides a forged answer that redirects the client to an attacker-controlled or malicious destination.

OSPF Future Work

OSPF: Planned Future Extension

OSPF is included as a planned extension rather than a completed core protocol in this project version. The intended future implementation would demonstrate neighbour formation, route advertisement, best path selection, link failure response, and possible abnormal routing behaviour. For the final submitted scope, the project focuses on ARP, DHCP, and DNS because these three protocols have been implemented in the website and supported with Ubuntu practical testing and Wireshark evidence.

Interactive Testing

What Users Can Test

The simulator allows users to test both guided and manual protocol scenarios. Predefined demonstrations show complete normal or attack sequences without requiring user input, while manual testing allows users to enter source and destination IP addresses and check whether the selected path is technically valid.

Normal Behaviour

Normal tests show how ARP, DHCP, and DNS are expected to work when devices and services behave correctly. These simulations focus on valid protocol operation, correct packet direction, and expected output messages.

Attack Behaviour

Attack tests show how trust weaknesses in local network protocols can be abused. ARP spoofing redirects traffic through an attacker, rogue DHCP provides malicious configuration, and rogue DNS returns a forged destination address.

Manual Validation

Manual validation prevents unrealistic testing by checking whether the entered source and destination pair is valid for the selected protocol. Invalid IPs, wrong-branch paths, unsupported attack paths, and incorrect source types are blocked.

Reset and Re-Test

The reset function stops the current animation, clears packet objects, removes line highlights, resets output logs, re-enables inputs, and returns the page to a ready state for the next test. This makes the simulator suitable for repeated live demonstration.

Assessment Alignment

How This Project Demonstrates Advanced Learning

This project demonstrates advanced learning by connecting network architecture, protocol behaviour, cybersecurity attack analysis, and evidence-based validation in one practical artefact. The enterprise topology shows how devices, routers, switches, clients, attackers, and services interact in a structured network. The protocol simulations explain how normal packet flows differ from abnormal or malicious behaviour. The Ubuntu and Wireshark evidence supports the analysis by showing that the demonstrated behaviours can be reproduced and verified at packet level.

This makes the project more than a visual website because it connects design, implementation, attack behaviour, and technical investigation. It directly supports the final presentation by allowing the project to be explained through three connected outputs: the website simulator, the Ubuntu practical demonstrations, and the packet capture evidence.
