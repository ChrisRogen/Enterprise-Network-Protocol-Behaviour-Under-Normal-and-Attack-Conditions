What DNS Does

DNS resolves domain names into IP addresses so users and applications can access services without remembering numeric addresses. In normal behaviour, a client sends a query to a trusted DNS server, and the DNS server replies with the correct IP address for the requested domain.

Why DNS Can Be Attacked

DNS can be abused when a rogue responder provides a forged answer before or instead of the legitimate DNS response. If the client accepts the forged answer, it may connect to an attacker-controlled IP address. This can support phishing, redirection, traffic interception, or malware delivery.

How This Page Proves It

The page demonstrates DNS using visual packet animation, validation logic, event logs, and Wireshark evidence. Normal behaviour shows a client query and legitimate server response. Attack behaviour shows the client being influenced by a rogue DNS response that returns a fake destination address.

Protocol Understanding

Normal DNS Resolution vs Rogue DNS Behaviour

DNS is essential for translating domain names into IP addresses, but the trust placed in DNS answers can be abused. A forged DNS response can redirect a user away from the legitimate destination and toward an attacker-controlled service.

Normal DNS Resolution

In normal DNS behaviour, a client sends a DNS query asking for the IP address of a domain. The legitimate DNS server responds with the correct IP address. The client can then connect to the real destination using the resolved address.

Rogue DNS Response

In rogue DNS behaviour, an attacker provides a forged DNS answer. If the client accepts the forged response, the requested domain can resolve to an attacker-controlled IP address instead of the legitimate service.

Normal vs Attack Difference

The key difference is answer integrity. In the normal path, the DNS response comes from the expected resolver and points to the correct destination. In the attack path, the response points to a false address, which can support phishing, redirection, or traffic interception.

Security Analysis

What the Attack Shows

The rogue DNS scenario shows that name resolution can become a redirection point for attackers. If a client accepts a forged DNS answer, the user may believe they are visiting a legitimate service while actually connecting to an attacker-controlled destination. This makes DNS abuse relevant to phishing, credential theft, malware delivery, and traffic interception.

Defensive Thinking

How Rogue DNS Risk Can Be Reduced

Practical mitigations include using trusted DNS resolvers, DNSSEC where supported, endpoint protection, network segmentation, DHCP snooping to prevent malicious DNS server assignment, monitoring for unexpected DNS servers, and Wireshark investigation using filters such as dns or udp.port == 53.

Reproduction Package

Recreate the DNS Practical on Another Local System

Open DNS Google Drive Folder Open DNS Instruction Manual PDF

The DNS reproduction package should contain the DNS packet capture files, the OVA file, and the instruction manual required to recreate the practical environment. The instruction manual PDF provides the full step-by-step process from importing the Ubuntu OVA into VirtualBox through to running MiniEdit/Mininet, capturing DNS traffic in Wireshark, executing normal DNS query-response testing, running rogue DNS response testing, saving evidence, and comparing the practical output with the website simulation.

1

Download Artefacts

Open the DNS Google Drive folder and download the OVA file, DNS PCAP files, and the instruction manual. Keep all files together in one local project folder so the VM, packet captures, screenshots, and recreation steps remain organised.

2

Read the Instruction Manual

Open dns-practical-recreation-instruction-manual.pdf before starting the practical. Follow the manual in order because it explains the required software, OVA import process, topology setup, Wireshark filters, normal DNS commands, and rogue DNS testing commands.

3

Import the Ubuntu OVA

In Oracle VirtualBox, choose File → Import Appliance, select the DNS OVA file, and complete the import. After import, start the Ubuntu VM and confirm that the desktop, terminal, MiniEdit/Mininet, and Wireshark are available.

4

Run the Normal DNS Test

Use MiniEdit/Mininet to start the DNS topology. Start the legitimate DNS service, send a DNS query from the client to the trusted DNS server, and capture the normal query-response exchange in Wireshark using filters such as dns or udp.port == 53.

5

Run the Rogue DNS Test

Follow the attack section in the manual to start a rogue DNS responder from the attacker host. Capture the forged DNS response in Wireshark and compare the returned IP address against the legitimate DNS response.

6

Compare with Website

Return to this DNS page and run the predefined normal and attack simulations. Compare the website packet animation and event logs with the Wireshark DNS evidence so the website, practical lab, and packet captures are clearly connected.
