Address Resolution Protocol Behaviour in an Enterprise LAN

Address Resolution Protocol is used inside a local network to map an IPv4 address to a Layer 2 MAC address. This ARP page demonstrates how legitimate ARP request and ARP reply messages allow hosts to communicate normally, and how the same trust-based behaviour can be abused when an attacker sends forged ARP replies.

The page connects three evidence layers: an interactive website simulation, a practical Ubuntu MiniEdit/Mininet lab, and Wireshark packet captures. The simulation allows guided and manual ARP testing, while the evidence section shows the real ARP request, legitimate ARP reply, spoofed ARP reply, and traffic redirection observed during the practical lab.

What ARP Does

ARP allows a device to discover the MAC address associated with an IPv4 address on the same local network. When a host wants to send traffic to another local host or to its default gateway, it checks its ARP cache first. If no valid entry exists, it broadcasts an ARP request such as “Who has this IP address?” and waits for the correct device to reply with its MAC address.

Why ARP Can Be Attacked

ARP does not include strong authentication in its basic operation. A malicious host can exploit this by sending forged ARP replies that claim the attacker MAC address belongs to another device, such as the default gateway. When the victim accepts the false mapping, traffic can be redirected through the attacker, enabling man-in-the-middle observation, manipulation, or denial of service.

Protocol Understanding

Normal ARP Behaviour vs ARP Spoofing

These diagrams explain how the visual simulator should be interpreted. Normal ARP builds a correct mapping between IP and MAC addresses. ARP spoofing abuses this process by inserting a false mapping into the victim's ARP cache.

Normal ARP Resolution

In normal ARP behaviour, the victim broadcasts an ARP request asking which device owns a specific IP address. The legitimate destination or gateway replies with its MAC address. After receiving this response, the victim can send Ethernet frames to the correct MAC address.

ARP Spoofing Attack

In ARP spoofing, the attacker sends forged ARP replies claiming that the attacker MAC address belongs to another IP address, such as the default gateway. If the victim accepts the forged reply, future traffic can be sent to the attacker instead of the legitimate device.

Normal vs Attack Comparison

The comparison view shows the key security difference. In the normal path, the victim learns the correct MAC address. In the attack path, the victim learns a malicious mapping and traffic is redirected through the attacker, creating a man-in-the-middle opportunity.

Security Analysis

What the Attack Shows

The ARP spoofing scenario demonstrates that a local attacker can exploit the trust relationship between IP and MAC address resolution. Once the victim stores a poisoned ARP entry, traffic that should go to a legitimate host or gateway may instead be sent to the attacker. This can allow observation, manipulation, or forwarding of traffic depending on how the attacker configures packet forwarding.

Defensive Thinking

How ARP Spoofing Can Be Reduced

Practical mitigations include switch-level controls such as Dynamic ARP Inspection, DHCP snooping, port security, static ARP entries for critical systems, network segmentation, endpoint detection alerts for duplicate IP-to-MAC changes, and Wireshark monitoring for repeated unsolicited ARP replies. These controls reduce the likelihood that a forged ARP response will be accepted silently.

Reproduction Package

Recreate the ARP Practical on Another Local System

Open ARP Google Drive Folder Open ARP Instruction Manual PDF

The linked Google Drive folder contains the ARP packet capture files and the OVA file required to recreate the practical environment. The instruction manual PDF provides the full step-by-step process from importing the Ubuntu OVA into VirtualBox through to running MiniEdit/Mininet, capturing ARP traffic in Wireshark, executing normal ARP testing, running ARP spoofing commands, saving evidence, and comparing the practical output with the website simulation.

1

Download Artefacts

Open the ARP Google Drive folder and download the OVA file, ARP PCAP files, and the instruction manual. Keep all files together in one local project folder so the evidence, VM, and recreation steps remain organised.

2

Read the Instruction Manual

Open arp-practical-recreation-instruction-manual.pdf before starting the practical. Follow the manual in order because it explains the required software, OVA import process, topology setup, Wireshark filters, normal ARP test commands, and ARP spoofing commands.

3

Import the Ubuntu OVA

In Oracle VirtualBox, choose File → Import Appliance, select the ARP OVA file, and complete the import. After import, start the Ubuntu VM and confirm that the desktop, terminal, MiniEdit/Mininet, and Wireshark are available.

4

Run the Normal ARP Test

Use MiniEdit/Mininet to start the ARP topology. Clear the ARP cache, run a ping from the victim host to the legitimate host, and capture traffic in Wireshark using filters such as arp or arp or icmp. Confirm the ARP request and legitimate ARP reply.

5

Run the ARP Spoofing Test

Follow the attack section in the manual to run the ARP spoofing command from the attacker host. Capture the forged ARP reply in Wireshark and confirm that the attacker is claiming ownership of another device's IP address.

6

Compare with Website

Return to this ARP page and run the predefined normal and attack simulations. Compare the website packet animation and event logs with the Wireshark evidence so the website, practical lab, and packet captures are clearly connected.
